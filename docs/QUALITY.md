# 质量策略

## 反馈层

1. `scripts/check-harness.ps1`：仓库地图、隐私路径、架构导入、文件预算和模板哈希。
2. Python `unittest`：周次、配置、匹配和输出构建等确定性逻辑。
3. Maven tests：权限、服务和工具类行为。
4. Vitest：前端 API client、token 生命周期、OAuth 回调和敏感错误处理。
5. Vite build：前端生产构建。
6. GitHub Actions：在干净环境重复同一验证入口。

统一入口：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/verify.ps1
```

## 测试优先级

1. 部门、人员、userid 数据权限和下载权限。
2. 上一完整周、周四补交截止、相邻归属窗口、应交/实交匹配、重名与异常提交。
3. 模板合规和管理筛选维度分离。
4. 任务进程的路径、超时、错误和敏感输出处理。
5. 身份认证、钉钉绑定和账号停用。
6. Markdown 报告解析与状态标签展示。

## 复杂度棘轮

已知热点通过 `harness/quality-baseline.json` 暂时豁免默认预算。正常功能开发应顺手抽取并下调预算，不进行一次性大爆炸重写。

## 当前技术债

| 债务 | 风险 | 下一步 |
| --- | --- | --- |
| 后端与 Python 原本零测试 | 权限或缺交统计回归难发现 | 为每个触碰的用例增加确定性测试 |
| `App.vue` 和 `styles.css` 过大 | 页面修改相互干扰 | 已抽取认证/API/登录切片；继续按周次、AI评价、任务、用户管理拆分，样式后续随功能切片迁移 |
| `FeedbackServiceImpl` 过大 | 通知、落盘和环境读取混杂 | 提取通知适配器和持久化边界 |
| 默认开发凭据存在 | 部署误用可能暴露数据 | 已加入显式开发模式、生产启动校验和 Compose 必填变量；继续审查历史固定账号引导逻辑 |
| 原始周报以文件保存 | 访问控制和生命周期依赖路径 | 明确保留策略并逐步建立审计边界 |
| 前端单包超过 1 MB | 首屏与缓存成本较高 | 功能拆分后引入懒加载和分包 |
