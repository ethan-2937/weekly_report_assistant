# 质量策略

## 反馈层

1. `scripts/check-harness.ps1`：仓库地图、隐私路径、架构导入、文件预算和模板哈希。
2. Python `unittest`：周次、配置、匹配和输出构建等确定性逻辑。
3. Maven tests：权限、服务和工具类行为。
4. Vitest：前端 API client、token 生命周期、OAuth 回调、敏感错误处理、评价折叠交互、授权姓名跳转、单人原文抽屉和 XLSX 导出结构。
5. Vite build：前端生产构建。
6. GitHub Actions：在干净环境重复同一验证入口。

统一入口：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/verify.ps1
```

## 测试优先级

1. 部门、人员、userid 数据权限和下载权限。
2. 单人周报原文的服务层范围过滤、未认证/越权拒绝、`report_id`/userid 匹配、敏感内容清洗和响应体上限。
3. 上一完整周、周四补交截止、周日当前周提醒窗口、相邻归属窗口、应交/实交匹配、免交名单、负责人履职证据、重名与异常提交。
4. 负责人下属 userid 映射优先级、部门回退候选、免交排除和未交下属导致的负责人履职风险。
5. 私有负责人覆盖的唯一 selector 解析、非负责人撤销、自映射/多人归属拒绝和环境净化。
6. 模板合规和管理筛选维度分离。
7. 项目明细字段提取、多值保留、旧周次空状态、服务层范围过滤和安全 XLSX 单元格。
8. 任务进程的路径、超时、错误和敏感输出处理；自动通知必须覆盖成功、无候选、失败、无效接收人和防重复路径。
9. 身份认证、钉钉绑定和账号停用。
10. Markdown 报告解析与状态标签展示。
11. 周一个人评价通知的已提交 userid 覆盖、私有反馈隐私校验、逐人发送、管理员结果和防重复路径。
12. 服务器 Codex Harness 的输入指纹、重试上限、隔离命令、报告章节/人员覆盖和敏感输出拒绝。

## 复杂度棘轮

已知热点通过 `harness/quality-baseline.json` 暂时豁免默认预算。正常功能开发应顺手抽取并下调预算，不进行一次性大爆炸重写。

## 当前技术债

| 债务 | 风险 | 下一步 |
| --- | --- | --- |
| 后端与 Python 原本零测试 | 权限或缺交统计回归难发现 | 为每个触碰的用例增加确定性测试 |
| `App.vue` 和 `styles.css` 过大 | 页面修改相互干扰 | 已抽取认证/API/登录切片；继续按周次、AI评价、任务、用户管理拆分，样式后续随功能切片迁移 |
| 反馈落盘仍位于 `FeedbackServiceImpl` | 文件日志失败不影响反馈，但审计边界仍较重 | 通知传输和接收人解析已抽出；后续单独提取反馈持久化边界 |
| 默认开发凭据存在 | 部署误用可能暴露数据 | 已加入显式开发模式、生产启动校验和 Compose 必填变量；继续审查历史固定账号引导逻辑 |
| 原始周报以文件保存 | 访问控制和生命周期依赖路径 | 明确保留策略并逐步建立审计边界 |
| 负责人附件依赖钉钉 Drive 权限 | 下载失败会导致履职内容无法判断 | 已加入官方接口受控下载、类型/大小限制和失败降级；持续监测权限与文件生命周期 |
| 前端单包超过 1 MB | 首屏与缓存成本较高 | 功能拆分后引入懒加载和分包 |
| Codex 评价依赖服务器 CLI 与模型额度 | 自动任务可能因登录、版本或额度失败 | 预检 CLI/Skill，失败保留上一份有效报告并记录安全错误码 |
