# Task: 周一个人评价暖心感谢

## 问题

周一 12:00 私人评价已有表扬、改进和 HR 联系方式，但结尾缺少温暖收束。需要按员工本人当周贡献生成两句式感谢，并用“团队因您”表达团队价值，同时不能增加周一模型调用或扩大隐私范围。

## 范围

- 包含：私有反馈 v2 `thanks` 字段、Skill/Harness 规则、Python 与 Spring 校验、通知模板、v1 回退和试发文案。
- 不包含：前端页面、管理评价正文、通知对象、发送时间、权限或幂等规则变化。

## 验收标准

- [x] 感谢语以“感谢您”开始，并含第二句“团队因您”。
- [x] 优先感谢本人当周有证据的具体贡献，证据不足时使用批准的固定暖心句。
- [x] 感谢语位于改进建议之后、HR 联系方式之前。
- [x] v1 旧反馈继续可发送，不触发重复通知或额外 Codex 调用。
- [x] 姓名、其他人员、客户/项目标识、userid、链接、路径和凭据不进入感谢语。

## 约束

- 相关产品不变量：只向已提交且稳定 userid 匹配的本人私发；管理员结果无人员正文。
- 相关架构边界：Codex 同次生成私有内容，Python 校验落盘，Spring 校验后发送。
- 需要保留的无关工作区修改：无。

## 验证

- 测试：Python 70 项、Spring 90 项、Vue 49 项全部通过；覆盖 v2 字段、温暖句式、隐私拒绝、v1 回退、消息顺序、试发和幂等。
- 手工/集成证据：Harness 检查和前端生产构建通过；只使用虚构员工、交付物和 userid。
- 统一命令：`powershell -ExecutionPolicy Bypass -File scripts/verify.ps1`

## 交付

- 决策：内容定制优先，固定暖心句只作旧数据或证据不足兜底。
- 剩余风险：模型措辞仍可能趋同；不为追求每周不同而制造无证据的新颖表达。
- 延后工作：无。
