# Task: 周日至周二自动采集与评价窗口

## 问题

现有服务器 cron 在周一至周三刷新上一业务周，并在周四补交截止后再运行一次。业务希望改为周日 18:10 起每两小时自动采集并按输入变化评价，持续到周二结束；若只修改 cron，周日运行会错误选择上一 ISO 周。

## 范围

- 包含：Harness 的计划窗口周次解析、Linux cron 示例、边界回归测试和部署文档。
- 不包含：周报提交归属窗口、周日提醒、周一个人评价通知、Web 手工采集入口。

## 验收标准

- [x] 周日 18:10、20:10、22:10 处理当前 ISO 周。
- [x] 周一和周二从 00:10 起每两小时处理同一个目标周。
- [x] 周三 00:00 起计划窗口安全跳过；显式 `--week-label` 和默认上一周模式保持可用。
- [x] 输入无变化时继续沿用指纹跳过 Codex，不增加无效模型调用。

## 约束

- 相关产品不变量：提交仍按周四至下一周周四前归属，自动评价截止不改变数据归属。
- 相关架构边界：cron 只调用 Python Harness，Spring 通知定时任务不承担采集或评价。
- 需要保留的无关工作区修改：无。

## 验证

- 测试：Python 67 项、Spring 89 项、Vue 47 项全部通过；覆盖计划窗口的起止边界、连续目标周和 ISO 年边界。
- 手工/集成证据：Harness 检查通过，前端生产构建通过，并核对 cron 表达式和 `--scheduled-window` 文档。
- 统一命令：`powershell -ExecutionPolicy Bypass -File scripts/verify.ps1`

## 交付

- 决策：用 Harness 解析目标周，不依赖 cron 日历推断上一周。
- 剩余风险：周二 22:10 后提交的数据不会自动进入评价，需显式周次人工重跑。
- 延后工作：如业务恢复周三补交自动刷新，再扩展计划窗口并补测试。
