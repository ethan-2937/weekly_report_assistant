# Task: 修复概览交互、AI 重点顺序与负责人证据链

## 问题

提交概览未提交卡被相邻卡片遮挡且跳跃动画丢失；点击后错误地持久化“未提交”筛选，导致已提交人员不可见。AI 评价的本周重点、红榜、黑榜顺序及黑榜颜色不符合管理阅读预期。钉钉部门推断出的负责人关系需要由用户确认的私有映射覆盖，并需要诊断一名负责人的附件解析失败原因。

## 范围

- 包含：统计卡层级与动效、状态页默认全量列表、AI 重点顺序与红黑榜颜色、私有负责人覆盖配置、附件解析失败诊断及相关回归测试。
- 不包含：后端五维评价契约、真实周报正文入库、公开仓库中的真实姓名/userid 配置、钉钉线上模板修改。

## 验收标准

- [ ] 未提交卡默认完整可见，桌面端恢复轻柔跳跃动画，移动端不重叠且尊重减少动态效果偏好。
- [ ] 点击未提交卡进入提交状态页后，已提交和未提交人员均显示，未提交人员仍排在前面。
- [ ] 本周重点位于 AI 红榜之前，AI 红榜位于 AI 黑榜之前；黑榜使用危险色而不是绿色。
- [ ] 私有负责人覆盖通过精确唯一 selector 解析为 userid；歧义、缺失、自映射或多人重复归属失败关闭。
- [ ] 真实负责人关系只进入服务器私有环境配置；代码、测试和文档示例仅使用虚构人员。
- [ ] 附件失败阶段得到可复核诊断；若属于通用代码问题，使用虚构附件补测试。

## 约束

- 相关产品不变量：后端与 Skill 继续保留正式五维评价；前端仅改变顺序和视觉语义。
- 相关架构边界：人员关系在 Python 采集层按 userid 固化，Vue 不猜测负责人；附件仍只进入忽略的目标周私有目录。
- 需要保留的无关工作区修改：任务开始时工作区干净，保留提交 `5b5a785`。

## 验证

- 测试：Vitest 覆盖跳转全量列表、卡片层级/动效、重点顺序与黑榜色彩；Python 覆盖负责人覆盖解析和附件失败路径。
- 手工/集成证据：桌面与移动端浏览器检查、附件安全状态检查、Vite 构建、Harness、`git diff --check`。
- 统一命令：`powershell -ExecutionPolicy Bypass -File scripts/verify.ps1`
- 结果：负责人覆盖、采集集成与 Harness 定向检查 21 项通过；前端相关 12 项通过，浏览器以虚构数据验证桌面和 390px 布局，控制台无错误。
- 结果：完整 `scripts/verify.ps1` 按用户要求不再重复执行，交付后由用户人工验证。
- 结果：目标周附件产物早于下载功能生成，DOCX 没有本地路径供 Skill 解析；同记录 HTML 被安全白名单拒绝属于预期行为。

## 交付

- 决策：负责人覆盖使用服务器私有单行 JSON；selector 精确唯一解析为 userid 后才参与统计、附件下载和履职评价。
- 决策：不开放 HTML 附件；重新采集目标周，让已支持的 DOCX 进入受控下载和解析链路。
- 剩余风险：未按用户要求重复执行完整统一验证；服务器重采集仍可能受 DingTalk Drive 权限或临时下载信息影响。
- 延后工作：部署后人工核对负责人私有映射和重采集后的脱敏附件状态。
