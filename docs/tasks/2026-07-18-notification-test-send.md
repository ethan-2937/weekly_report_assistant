# Task: 管理员向单一确认接收人试发两类自动通知

## 问题

周日未交提醒和周一个人评价通知只能等待定时器真实触发，现有代码没有安全的单人试发入口。直接缩短 cron 会向真实候选人员发送，并可能写入正式幂等状态，无法用于上线前验证。

## 范围

- 包含：管理员专用试发接口、明确接收人二次确认、周日批量发送链路样例、周一个性化发送链路样例、运行状态页按钮和回归测试。
- 不包含：读取真实周报、生成真实评价、修改正式 cron、清除或写入周日/周一正式幂等状态。

## 验收标准

- [x] 只有 ADMIN 可以试发，未认证返回 401，其他角色返回 403。
- [x] 接收人名称必须与服务器反馈接收人精确一致，且只能解析到一个 userid。
- [x] 周日样例使用 `sendMarkdown`，周一样例使用 `sendPersonalMarkdown`。
- [x] 消息醒目标注“测试”，不声称真实未交或真实评价。
- [x] 响应和日志不返回 userid、token、AppSecret 或钉钉原始响应。
- [x] 试发不读取候选数据，也不写正式通知状态文件。

## 约束

- 相关产品不变量：正式任务继续按周日当前周和周一上一周运行；远程结果不确定时不自动重发。
- 相关架构边界：Controller 依赖 Service；通知传输复用现有钉钉适配器；前端只提供管理员确认入口。
- 需要保留的无关工作区修改：无。

## 验证

- 测试：服务双发送链路、接收人失败关闭、Web 认证/授权、前端 API 与按钮行为。
- 手工/集成证据：服务器管理员点击后由用户确认张艺政收到两条测试消息。
- 统一命令：`powershell -ExecutionPolicy Bypass -File scripts/verify.ps1`
- 结果：2026-07-18 完整验证通过；Python 63 项、Spring 80 项、Vue 45 项，前端生产构建成功。

## 交付

- 决策：试发只验证接收人解析、钉钉凭据、应用可见范围和两种传输方式；正式候选及评价输入仍由各自定时任务验证。
- 剩余风险：测试成功不代表定时任务输入文件已就绪，仍需分别执行周日预检和周一评价准备检查。
- 延后工作：不提供任意 userid 试发或批量测试入口。
