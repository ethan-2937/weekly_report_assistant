# Task: 周四截止的上一周补交归属

## 问题

当前 `resolve_week_args` 同时把周报业务周期和钉钉提交查询窗口定义为周一至周日。员工在下一周周一至周三补交上一周周报时，报告会因提交时间落在下一个 ISO 周而被标记为上一周“未提交”。现场证据已确认：W28 周一补交的报告 userid 与模板均正确，但不在 W28 的周一至周日查询窗口内。

## 范围

- 包含：将周报业务周期保持为周一至周日，并把该周的提交归属窗口调整为本周周四 00:00 至下一周周四 00:00 前。
- 包含：默认上一周、当前周和显式日期参数的确定性边界测试。
- 包含：采集 API 时间参数、控制台输出、提交摘要和分析输入中的周期说明。
- 包含：产品、质量、架构、Skill、AGENTS 和 Harness 同步。
- 不包含：修改钉钉线上模板、周报正文、评价口径、权限模型或真实历史数据。
- 不包含：自动读取正文判断报告自述周期；归属只依赖明确的周四截止时间规则。

## 验收标准

- [x] W28 业务周期仍为 2026-07-06 至 2026-07-12。
- [x] W28 提交归属窗口为 2026-07-09 00:00 至 2026-07-15 23:59:59.999（Asia/Shanghai）。
- [x] 周三 23:59:59.999 的提交归上一周，周四 00:00 起归下一周。
- [x] 周一至周三补交不再漏记，且上一周周一至周三的提交不会重复计入本周。
- [x] `--start/--end` 继续表示业务周期，输出目录仍使用业务周期起始日的 ISO 周标签。
- [x] 自动化测试不读取 `.env`、真实通讯录、真实周报或调用钉钉。

## 约束

- 相关产品不变量：应交与实交仍优先按稳定 userid 匹配；业务周标签和输出路径保持兼容。
- 相关架构边界：日期/归属计算集中在 `dingtalk_common.py`；网络适配器只接收已计算的毫秒时间戳。
- 需要保留的无关工作区修改：开始时 Git 工作区干净；`output/` 中本次诊断数据保持忽略，不进入 Git。

## 验证

- 测试：Python 周次/提交窗口单元测试、既有后端与前端测试。
- 手工/集成证据：固定 W28 日期边界、周三/周四临界点、输出周期文案。
- 统一命令：`powershell -ExecutionPolicy Bypass -File scripts/verify.ps1`

## 交付

- 决策：周报业务周期仍按 ISO 周命名；提交采用周四到下周三的唯一归属窗口，为下一周周一至周三提供补交宽限期。
- 验证：8 个 Python 测试覆盖 W28 固定边界、相邻窗口无缝无重叠、`run_weekly.py` 与 `download_reports.py` 的适配器参数和输出文案；统一 Harness、27 个 Spring 测试、8 个 Vitest 与生产构建通过。
- 运行证据：经用户授权按新规则重建本机 W28 后，周一上午补交且 userid/模板匹配的报告已从“未提交”变为“已提交”；真实运行数据仅写入被 Git 忽略的 `output/`。
- 剩余风险：周四前提前提交“本周”周报会按规则归入上一周；这是用户明确接受的组织行为假设。
- 延后工作：如未来钉钉模板增加明确的“所属周次”字段，可改为字段优先、时间窗口降级。
