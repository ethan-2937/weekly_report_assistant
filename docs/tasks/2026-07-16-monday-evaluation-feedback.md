# Task: 周一 12:00 向已提交员工发送个人周报评价通知

## 问题

现有 Spring Scheduler 仅在显式启用时于每周日 18:00 检查当前业务周，并向未提交人员发送私人提醒。正式 Codex 评价在上一业务周采集完成后写入 `manager_report.md`，但没有稳定、机器可读的个人反馈清单，也没有周一评价通知任务。

不能在发送时从 Markdown 姓名猜测收件人，也不能为每名员工重新调用一次 Codex。需要让现有一次 Codex 评价同时生成私有结构化反馈，并由 Harness 按提交状态和稳定 `userid` 校验后落盘；Spring 只消费已验证结果。

## 范围

- 包含：
  - Codex 结构化输出增加每位已提交员工的“做得好的地方”和“重点改进”。
  - Python Harness 校验反馈人员覆盖、稳定 `userid`、字段长度、敏感内容和目标周，并原子写入受限输出目录。
  - 默认关闭的 Spring 周一 12:00 定时任务，按稳定 `userid` 逐人发送私人钉钉工作通知。
  - 通知末尾使用服务器私有环境变量配置的 HR 联系人，不把真实姓名写入仓库。
  - 任务成功或失败时向现有管理员接收人发送不含姓名、userid 和评价正文的汇总结果。
  - 防重复状态、发送结果不确定时不自动重发、配置和部署文档。
- 不包含：
  - 修改五维评价标准、周报模板或负责人规则。
  - 周一再次调用 Codex、按姓名猜测 userid、群发包含多人评价的消息。
  - 在测试或开发过程中连接真实钉钉、真实 MySQL 或真实员工数据。

## 验收标准

- [x] 现有一次 Codex 评价为全部且仅限已提交员工生成私有结构化反馈。
- [x] 反馈缺人、多出人员、重复 userid、空内容、超长内容、敏感字段或周次不一致时 Harness 拒绝落盘。
- [x] 启用后每周一 12:00（Asia/Shanghai）读取上一完整业务周的成功评价并逐人私发一次。
- [x] 每条员工消息先展示做得好的地方，重点展示需要改进的地方，并以配置的 HR 联系方式结尾。
- [x] 未提交、免交或反馈未通过校验的人员不接收评价通知。
- [x] 配置缺失、评价未就绪、人员不一致或发送异常时失败关闭，并向管理员发送安全汇总。
- [x] 已完成周次不重复发送；远程发送结果不确定时不自动重发。
- [x] 状态文件不保存姓名、userid 或评价正文；日志不输出人员、正文、token 或原始钉钉响应。
- [x] 功能默认关闭，所有测试使用虚构姓名和 userid，且不访问真实外部服务。

## 约束

- 相关产品不变量：上一完整 ISO 周按周一至周日命名；人员和通知收件人使用稳定 `userid`；周一评价是个人反馈，不改变正式管理评价。
- 相关架构边界：Codex Skill 负责评价措辞；Python Harness 负责覆盖/隐私校验与原子落盘；Spring 负责定时编排、幂等和钉钉通知。
- 需要保留的无关工作区修改：未跟踪的 `docs/MULTI_AGENT_COORDINATOR_PROMPT.md` 不读取、不修改、不暂存。

## 验证

- 测试：Python 定向 15 项、Spring 定向 14 项通过；统一验证中 Python 58 项、Spring 63 项、Vitest 34 项通过。
- 手工/集成证据：Harness 检查、示例配置与 Compose 变量一致性、前端生产构建通过；`git diff --check` 和暂存区隐私扫描在提交前执行。
- 统一命令：`powershell -ExecutionPolicy Bypass -File scripts/verify.ps1`

## 交付

- 决策：个人反馈与正式报告在同一次 Codex 调用中生成；反馈文件只作为服务器受限中间产物；周一任务不解析自然语言 Markdown。
- 剩余风险：周一 12:00 前 Codex 自动评价必须已经成功；首次启用前需在服务器私有配置中填写 HR 联系人、重新复制 Skill 并重新生成目标周评价；生产构建仍有既有的 VueUse 注解和主包大小警告。
- 延后工作：如需人工重发，应另建带审计和明确确认的管理入口，不复用自动任务绕过幂等保护。
