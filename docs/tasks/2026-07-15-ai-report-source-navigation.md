# Task: AI 评价中的姓名可按权限查看对应周报原文

## 问题

AI 评价当前由 `MarkdownReport.vue` 统一渲染 Markdown，但姓名只是普通文本；评价页已经同时加载当前周的授权提交状态，其中包含姓名、稳定 `userid` 和 `report_id`，后端原始周报位于受限的 `raw/reports.json`，现有模板合规服务已经具备按 `report_id`、`creator_id` 匹配报告的代码证据。

直接在前端预加载所有原文会扩大敏感数据暴露面和响应体积；只依赖姓名也会在重名时误跳。需要建立授权人员索引、单人按需接口和服务层二次权限校验。

## 范围

- 包含：
  - AI 评价页中表格、列表、段落、标题等区域的授权人员姓名交互。
  - 悬停/键盘聚焦提示和单人周报原文抽屉。
  - Spring 服务层按周次和稳定 `userid` 返回裁剪后的单人周报模板字段。
  - 无权限、未认证、未提交、原文缺失和重名路径的回归测试。
  - 前后端接口、安全、质量文档与复杂度预算同步。
- 不包含：
  - 修改 Codex 评价口径、Skill、周报模板或现有正式报告内容。
  - 返回原始 JSON、附件、内部路径、token 或其他人员正文。
  - 引入 Vue Router、全局状态框架或数据库结构变更。
  - 改变提交概览、未交名单、用户管理、登录等其他前端界面。

## 验收标准

- [x] AI 评价中能够与当前账号授权名单稳定匹配的姓名均可点击，悬停显示查看原文提示。
- [x] 点击姓名仅按需请求选定周次、选定人员的原文，并在评价页抽屉内按模板字段展示。
- [x] 接口不批量返回全员正文或附件；单份预览超过受控文本上限时安全拒绝，避免 `payload too large`。
- [x] 同名人员不被猜测；用户需按虚构部门/职务选择稳定身份。
- [x] 未提交或原文缺失时展示安全空状态，错误不包含正文、userid、token 或内部路径。
- [x] 后端在服务层再次执行当前账号数据范围过滤；范围外人员不可通过直接调用接口读取。
- [x] 未认证请求被拒绝，范围外请求不确认人员是否存在。
- [x] `MarkdownReport` 的默认行为不变，提交概览等非 AI 评价使用场景不启用姓名交互。
- [x] 测试全部使用明显虚构姓名、部门和 userid，且不访问真实网络、钉钉或 MySQL。

## 约束

- 相关产品不变量：原始周报仅从目标周受限文件读取；人员优先按稳定 `userid` 匹配；后端权限决定可见数据。
- 相关架构边界：Vue 只负责呈现；API 调用复用统一 client；Controller 不直接读文件；Spring 服务层过滤后再由 mapper/解析边界读取原文。
- 需要保留的无关工作区修改：未跟踪的 `docs/MULTI_AGENT_COORDINATOR_PROMPT.md` 不读取、不修改、不暂存。

## 验证

- 测试：后端定向 15 项通过；统一验证中 Spring 53 项、Python 56 项、Vitest 33 项通过。
- 手工/集成证据：前端生产构建通过；`App.vue` 1120 行、`MarkdownReport.vue` 351 行，热点预算按实际值下调；`git diff --check`、隐私扫描和工作区状态在提交前执行。
- 统一命令：`powershell -ExecutionPolicy Bypass -File scripts/verify.ps1`

## 交付

- 决策：使用授权提交状态构建人员索引；仅在 AI 评价页启用；单人原文懒加载且不预取/批量缓存；服务层使用稳定 `userid` 校验并按 `report_id` 优先匹配，同时限制单份预览文本总量。
- 剩余风险：AI 正文中不属于授权周次名单的别名无法安全自动链接；原始附件展示不在本次范围；生产构建仍有既有的 VueUse 注解和主包大小警告。
- 延后工作：如未来需要跨周别名或附件预览，应建立显式身份元数据和独立附件授权接口。
