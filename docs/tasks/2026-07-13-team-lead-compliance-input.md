# Task: Make team-lead compliance evidence observable

## 问题

团队负责人履职检查为空。当前确定性输入只在未提交候选表中携带负责人标记，已提交报告块没有负责人身份、职务或团队；钉钉附件字段只保留元数据，空数组与真实附件没有被区分。W28 结构证据显示负责人能够从通讯录识别，但正式分析无法可靠建立负责人清单，也无法区分“无团队汇总”与“附件存在但正文未解析”。

## 范围

- 包含：为全部负责人生成确定性的履职输入表；解析附件元数据并区分无附件、疑似团队汇总和附件待解析；在已提交报告块补充负责人身份；范围账号按服务层权限过滤负责人输入和正式履职表；同步 Skill、输出格式、架构、质量和隐私文档。
- 不包含：调用未经确认的钉钉文件下载 API；解析真实 PDF/Word/Excel 附件正文；修改钉钉线上模板；重写前端 Markdown 渲染器；修改负责人名单来源。

## 验收标准

- [x] 所有负责人候选均进入负责人履职输入，个人周报已交与未交状态明确。
- [x] 已提交报告块明确包含负责人标记、部门和职务。
- [x] 附件值 `[]` 不得判为已提交附件；真实文件条目显示数量和安全文件名。
- [x] 有疑似团队汇总附件但正文未解析时显示“附件待解析”，不得误写“未见提交”。
- [x] 无个人周报、无团队汇总证据和正文履职证据分别给出稳定、非空状态。
- [x] 范围账号只能看到授权负责人的履职输入和正式履职表；`REPORT_ALL` 仍获得原始完整 Markdown。
- [x] 错误和测试输出不包含 token、附件下载凭据、真实人员或原始周报。

## 约束

- 相关产品不变量：负责人个人周报和团队额外职责分别评价；缺少证据时不能猜测合格或不合格。
- 相关架构边界：Python 生成确定性证据，Skill 生成管理结论，Spring 在服务层过滤，Vue 只渲染 Markdown。
- 需要保留的无关工作区修改：开始任务时工作区干净；上一任务提交 `fdb55e7` 已推送 Codeup `master` 与 GitHub `main`。

## 验证

- 测试：附件空数组、附件元数据、正文证据、未提交负责人、非负责人、范围过滤和完整权限回归。
- 手工/集成证据：使用临时目录和虚构 W28 输入生成分析包，不读取真实 `output/` 作为测试夹具。
- 统一命令：`powershell -ExecutionPolicy Bypass -File scripts/verify.ps1`

## 交付

- 决策：附件正文不可用时保留“附件待解析”中间状态，避免把技术不可见性解释为负责人未履职。
- 实现：`leader_compliance.py` 生成负责人证据，`weekly_outputs.py` 统一输出，`MarkdownTableFilter` 按负责人姓名/userid 精确过滤授权行；未授权行即使提及授权员工也不会被保留。
- 验证：20 个 Python 测试、28 个 Spring 测试、8 个 Vitest 测试、前端生产构建、Harness 和 `scripts/verify.ps1` 全部通过。
- W28 离线证据：在临时目录生成 15 行负责人输入且 15 行状态均非空；8 个已提交报告可由通讯录稳定识别为负责人，未覆盖持久化 `output/`。
- 复杂度：`run_weekly.py` 从 282 行降至 134 行，预算同步降为 134；新增 Python/Java 文件均低于默认预算。
- 隐私：测试只使用虚构人员、userid、附件名和正文；日志与错误不输出 fileId、spaceId、下载地址或附件原始元数据。
- 剩余风险：只有附件元数据时无法判断整体进度、成员评价、风险和资源需求是否合格。
- 延后工作：确认钉钉文件下载 API、应用权限和文件生命周期后，增加受控附件下载与文本抽取适配器。
