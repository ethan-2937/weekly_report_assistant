# Task: Codex 自动评价输出权限修复

## 问题

Docker 采集以容器 root 在绑定挂载的 `output/` 中创建周目录，宿主机用户运行 Codex Harness 时可能无法创建 `automation/` 或原子更新 `summary/`。Skill 过期等预检错误还会尝试写失败状态，使权限异常覆盖原始安全错误码。

## 范围

- 包含：宿主机写目录预创建与探测、原子写父目录错误封装、预检失败路径和服务器修复说明。
- 不包含：读取或修改真实周报、改变容器认证、放宽目录到组外或所有用户。

## 验收标准

- [x] 预检失败只输出原始安全错误码，不写目标周状态。
- [x] Docker 采集前由宿主机创建并验证 `automation/` 和 `summary/` 可写。
- [x] 父目录创建失败转换为安全错误码，不暴露堆栈或内部文件内容。
- [x] 既有 root 属主目录有一次性最小权限修复说明。

## 约束

- 相关产品不变量：评价输入、正式报告路径、权限过滤和数据内容保持不变。
- 相关架构边界：容器负责采集，宿主机 Harness 负责校验和正式写入。
- 需要保留的无关工作区修改：无。

## 验证

- 测试：Python 70 项、Spring 89 项、Vue 47 项全部通过；覆盖输出目录创建、无残留探针、不可写路径和原子写失败封装。
- 手工/集成证据：Harness 检查和前端生产构建通过；不读取 `output/`，仅验证虚构临时目录。
- 统一命令：`powershell -ExecutionPolicy Bypass -File scripts/verify.ps1`

## 交付

- 决策：保持 Docker 采集模式，用宿主机预创建关键目录解决 UID 边界。
- 剩余风险：升级前已经由 root 创建的目录仍需在服务器执行一次属主修复。
- 延后工作：如容器后续改为非 root 用户，再统一 Compose UID/GID 配置。
