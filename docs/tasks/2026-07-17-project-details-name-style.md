# Task: 项目明细增加姓名并对齐参考表样式

## 问题

项目明细当前公开列表和 XLSX 只有七列，无法直接识别每条成本记录对应的填报人；导出虽然可用，但标题字体、蓝色表头、边框、列宽和行高与用户提供的 `周报项目明细表.xlsx` 仍有差异。

## 范围

- 包含：授权范围内的姓名列、前端搜索与表格、八列 JSON/XLSX 契约、参考表视觉样式和回归测试。
- 不包含：公开 userid/部门、读取真实周报正文、改变项目拆分或费用分摊规则。

## 验收标准

- [x] “姓名”位于“序号”之后，只随当前账号有权查看的项目明细返回和导出。
- [x] 前端项目明细展示姓名，并支持按姓名搜索。
- [x] XLSX 使用合并标题、参考蓝色表头、等线/宋体、黑色细边框和接近参考表的行列尺寸。
- [x] XLSX 仍保持数值类型和公式注入防护，下载接口和权限行为不变。
- [x] JSON 与 XLSX 不返回 userid、部门或其他隐藏身份字段。

## 约束

- 相关产品不变量：服务层权限过滤先于公开姓名；多个客户/项目不拆分，不推断费用分摊。
- 相关架构边界：Python CSV 保留权限字段；Spring 过滤并生成 XLSX；Vue 只展示授权结果。
- 需要保留的无关工作区修改：无。

## 验证

- 测试：Spring 过滤和八列 XLSX、Vue API/组件、完整回归套件。
- 手工/集成证据：参考工作簿和生成工作簿均通过渲染检查。
- 统一命令：`powershell -ExecutionPolicy Bypass -File scripts/verify.ps1`
- 结果：2026-07-17 完整验证通过；Python 63 项、Spring 74 项、Vue 40 项，前端生产构建成功。
- 视觉结果：生产导出预览与参考表对比通过；合并范围 `A1:H1`、表头色 `#84B7FE`，公式错误扫描为 0。

## 交付

- 决策：姓名属于本页面明确需要的最小展示字段；userid 和部门继续只用于服务层过滤。
- 剩余风险：过长的客户或项目文本仍需在 Excel 单元格中换行展示。
- 延后工作：不在下载文件中加入组织架构或人员标识列。
