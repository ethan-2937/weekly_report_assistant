# Task: 周日 18:00 自动提醒未交周报人员并通知管理员执行结果

## 问题

当前系统能够按稳定 `userid` 生成周报提交统计，也能够通过钉钉企业工作通知发送 bug 反馈，但没有周日自动检测和提醒。直接复用 `run_weekly.py --week current` 会写入未完成周的正式输出并下载负责人附件；直接扩展 `FeedbackServiceImpl` 又会继续混合反馈落盘、收件人解析和通知传输。

主要风险是：使用陈旧数据误提醒、遗漏免交规则、把周日提醒误称为最终逾期、重复调度造成骚扰、采集失败后按不完整名单发送，以及在日志或异常中暴露 userid、token 或钉钉响应。

## 范围

- 包含：实时计算当前业务周截至周日 18:00 的未交候选；复用现有免交规则和周四归属窗口；抽取通用钉钉工作通知适配器；增加默认关闭的 Spring 定时调度；向未交人员发送私人提醒；向配置的反馈接收人发送成功或失败结果；增加幂等状态和安全日志；同步 Docker、示例配置和部署文档。
- 不包含：前端页面、最终缺交认定、周报评价口径、负责人附件处理、Codex 分析、真实免交名单、真实通讯录或周报、历史 `web/backend/`。

## 验收标准

- [x] 周日 18:00 使用 `Asia/Shanghai` 和当前 ISO 业务周，只查询本周周四 00:00 至检测时刻。
- [x] 应交、实交和免交人员复用现有稳定 userid 与精确免交规则；免交和已交人员不会收到提醒。
- [x] 员工通知不包含其他人员信息；管理员在成功、无缺交和失败路径都收到只含周次、人数及安全状态的结果通知。
- [x] 模板、采集、配置或通知失败时不基于不完整数据继续发送员工提醒。
- [x] 同一周的成功任务不会重复提醒；不确定的远程发送状态不会自动重发员工通知。
- [x] 默认测试不访问钉钉、真实网络、真实 MySQL、真实配置或敏感输出目录。

## 约束

- 相关产品不变量：业务周为周一至周日，提交归属窗口从本周周四至下周周四前；周日提醒不是最终逾期结论；人员优先按稳定 userid 匹配；免交名单在比较前过滤。
- 相关架构边界：Python 负责实时采集和候选计算；Spring 负责任务调度、通知编排和通知适配；通知凭据沿用现有反馈通知配置，不写入仓库。
- 需要保留的无关工作区修改：开始时工作区干净；后续若出现非本任务改动则停止并确认。

## 验证

- 测试：Python 36 项通过，覆盖固定时钟、当前周窗口、userid 差集、免交、空模板、默认安全预检和错误清洗；Spring 45 项通过，覆盖通知适配、无效接收人、反馈回归、管理员成功/失败通知、员工提醒、空通讯录失败关闭和防重复状态。
- 手工/集成证据：Compose 使用虚构必填凭据执行 `config --quiet` 通过；`run_weekly.py` 从 134 行降至 88 行，`FeedbackServiceImpl` 从 593 行降至 334 行，质量预算同步只下调不提高；`git diff --check` 通过。
- 统一命令：`powershell -ExecutionPolicy Bypass -File scripts/verify.ps1`

## 交付

- 决策：通知采用钉钉企业工作通知；员工收到通用私人提醒，管理员收到独立汇总结果；功能默认关闭，部署核对后显式启用。统一验证已通过，包含 Harness、Python、Spring、Vitest 和前端生产构建。
- 剩余风险：钉钉应用可见范围和生产凭据只能在服务器以真实账号做最终联通验证；远程接口接受请求后进程立即崩溃无法实现严格的端到端 exactly-once。
- 延后工作：前端任务状态页、节假日跳过规则和多实例分布式锁。
