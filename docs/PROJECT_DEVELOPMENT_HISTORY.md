## 1. 这是什么文档

软件工程中没有一个唯一名称可以覆盖“从立项到当前的所有开发记录”。通常会把以下材料组合起来：

| 材料 | 主要回答的问题 | 本项目对应文件 |
| --- | --- | --- |
| 产品需求/产品契约（PRD） | 系统为谁解决什么问题，哪些规则不能变 | [`docs/PRODUCT.md`](PRODUCT.md) |
| 架构说明（Architecture Description） | 模块如何分工，数据和权限如何流动 | [`docs/ARCHITECTURE.md`](ARCHITECTURE.md) |
| 任务记录/迭代记录 | 一次变更为什么做、做了什么、如何验收 | [`docs/tasks/`](tasks/) |
| ADR（架构决策记录） | 重要技术选择和取舍是什么 | 本报告的“关键决策”章节与各任务记录 |
| 测试与质量记录 | 如何证明功能、失败路径和安全边界 | [`docs/QUALITY.md`](QUALITY.md)、`tests/`、Spring/Vitest 测试 |
| 安全与隐私记录 | 哪些数据能读、能写、能进日志和 Git | [`docs/SECURITY_PRIVACY.md`](SECURITY_PRIVACY.md) |
| 项目开发历程与工程实践报告 | 项目如何演进、发现了什么问题、形成了什么工程能力 | 本文 |

因此，本文是“项目总览和过程证据索引”，不是产品契约的替代品。未来继续开发时，应在对应任务记录和产品/架构文档中记录细节，再在本文追加阶段性结果。

## 2. 项目目标与范围

### 2.1 立项目标

项目最初要解决的是一个完整但容易出错的人工链路：从钉钉通讯录和周报中确认应交人员，统计提交情况，整理个人工作内容，再形成管理者可阅读的周报评价和负责人履职检查。

目标逐步收敛为一条可复核的自动化链路：

```text
钉钉通讯录/周报
    -> Python 确定性采集和匹配
    -> CSV、Markdown、负责人证据和分析输入包
    -> weekly-report-assistant Skill/Codex 生成评价
    -> Harness 校验覆盖、周次和隐私
    -> Spring 权限过滤和任务编排
    -> Vue 管理页面、下载和按需查看原文
    -> 周日提醒与周一个人反馈通知
```

### 2.2 不在范围内的内容

- 不把历史 `web/backend/` 继续扩展为新后端；当前功能进入 `web/backend-spring/`。
- 不修改钉钉线上周报模板镜像 `weekly_report_template.txt`，除非取得明确授权。
- 不把真实通讯录、原始周报、密钥、密码、访问令牌或生产附件放入代码仓库。
- 不让模型猜测人员身份、应交名单、负责人下属或缺失证据。

## 3. 发展时间线

### 阶段一：从采集脚本到可运行的 Skill（2026-06-26）

**触发问题。** 手工下载和整理周报的成本高，人员匹配容易被重名、提交时间和模板字段影响，模型评价也缺少稳定输入格式。

**主要工作。** 初始提交建立 Python 周报采集/输出链路；随后加入 `weekly-report-assistant` Skill、部署说明和 Youzhi 视觉品牌。Python 将钉钉通讯录、周报和提交状态转换成固定目录结构，Skill 负责在明确授权的输入上生成管理报告。

**效果。** 形成了“采集和规则由程序负责、自然语言评价由 Skill 负责”的第一版边界，避免把 API 调用、身份匹配和模型判断混在一个脚本中。

### 阶段二：部署稳定性和报告可读性（2026-06-26 至 06-30）

**发现的问题。** Docker 网络、服务器端口、提交字段映射和 Markdown 表格呈现会直接影响部署可用性和管理阅读；前端初期还存在过度突出的品牌卡片、宽屏布局和信息密度问题。

**主要工作。** 修复 Docker 网络和端口约定，切换到 Codeup 部署文档；校正提交状态字段映射；改进 Markdown 表格、报告排版、AI 评价页宽度和颜色规则；保留钉钉模板不可变保护。

**效果。** 系统可以在本地和 Ubuntu/Docker 环境按同一输出契约运行，管理报告从原始表格变成更易阅读的页面。模板内容和管理筛选维度开始明确分离。

### 阶段三：认证、账号和数据权限（2026-07-01 至 07-03）

**发现的问题。** 仅依赖页面隐藏不能保护员工周报；系统需要账号、JWT、钉钉 OAuth 和可管理的角色/范围。与此同时，管理员账号管理职责不应自动等同于完整周报阅读权限。

**主要工作。** 引入 MySQL 持久化认证表、BCrypt 密码、JWT、用户名密码登录、钉钉 OAuth 登录、反馈入口和账号管理界面；加入 `REPORT_ALL`、`ADMIN` 以及部门/人员/稳定 `userid` 范围。

**效果。** 后端服务层成为数据权限的唯一裁决点：`REPORT_ALL` 或显式 `ALL` 才能获得完整周报；`ADMIN` 可以管理账号和任务，但不会仅因角色自动获得完整周报；列表、详情、Markdown 和 CSV 使用同一过滤边界。

### 阶段四：风险优先的 Harness 和回归测试（2026-07-13）

**触发原因。** 早期功能已经较多，但后端权限、生产默认凭据、免交名单、负责人履职和前端认证都缺少系统化的机械验证；巨型 `App.vue` 把请求、token 和登录逻辑集中在一处，修改风险较高。

**主要工作。**

- 建立根目录和子目录 `AGENTS.md` 地图、产品/架构/质量/安全文档、任务模板、Maven Wrapper、GitHub Actions 和统一 `scripts/verify.ps1`。
- 以虚构数据补充权限回归：`REPORT_ALL`、`ADMIN`、`DEPT:`、`USER:`、`USERID:`、兼容普通文本、多范围、空范围、无匹配范围、列表/Markdown/CSV 下载和任务接口。
- 增加 Spring Security HTTP 状态测试，验证未认证为 401、权限不足为 403、错误不携带正文/token/密码。
- 增加生产凭据校验：生产模式拒绝默认 JWT secret 和默认管理员密码；显式开发模式仍支持本地快速启动；Docker 对生产变量使用必填插值。
- 从 `App.vue` 抽取 API client、认证状态 composable 和登录视图，建立 Vitest 基础，覆盖 Authorization、401、OAuth URL 清理和错误脱敏。

**效果。** 权限逻辑从“代码看起来正确”进入可回归验证；认证和 API 请求有明确边界；生产部署不会静默接受开发凭据。`App.vue` 按真实缩减下调预算，`styles.css` 因未安全拆分而保持原预算。

### 阶段五：周期、补交、免交和负责人证据（2026-07-13）

#### 周四补交归属

**问题。** 大家通常在周末写周报，周一至周三的补交不能被误判为新周提交，也不能让相邻周窗口重叠。

**方案。** 业务周继续按周一至周日的 ISO 周命名；提交归属窗口统一为本周周四 00:00 至下一周周四 00:00 前，时区固定为 `Asia/Shanghai`。边界测试覆盖周三 23:59:59.999 和周四 00:00。

**效果。** 以 W28 为例，业务周期仍是 2026-07-06 至 2026-07-12，补交窗口覆盖 2026-07-09 至 2026-07-15，解决了“实际补交但页面显示未提交”的一类时间归属问题。

#### 免交人员配置

**问题。** 免交名单写在宿主机环境后，Docker Compose 没有透传，或空的配置文件覆盖了容器环境；只在前端隐藏也会导致应交人数、已交人数和分析输入不一致。

**方案。** 采集层在生成确定性产物前过滤免交人员及其报告，优先使用 `USERID:`，其次使用精确的 `NAME:` 和兼容普通文本；不做模糊匹配，不自动推断。真实名单只保存在未跟踪服务器配置，仓库示例只保留规则。

**效果。** 统计、缺交名单和 Codex 输入使用同一人员集合，配置错误会安全失败而不是静默扩大或缩小范围。

#### 负责人履职证据

**问题。** 负责人履职区域曾经大面积为空，原因不一定是负责人“不合格”：早期输入只有负责人身份或附件元数据，没有正文证据；负责人下属关系也缺少稳定 `userid` 映射。

**调研和方案。** 先把“没有附件”“有附件但正文未解析”“有正文证据”“未提交个人周报”区分为不同状态，再接入受控的钉钉 Drive 下载信息接口。负责人覆盖使用服务器私有的精确唯一 selector；没有显式覆盖时，按共享钉钉部门生成待确认候选，不猜测最终关系。免交人员从负责人下属统计中排除。

**效果。** “证据不可见”不再被错误解释为“未履职”；负责人个人周报、团队额外职责、附件状态和下属缺交风险可以分别审计。下载限制为 HTTPS、受支持文件类型、单文件和每周总大小上限，分析包只保存本地相对路径。

### 阶段六：服务器 Codex Harness 与自动评价（2026-07-14 至 07-15）

**问题。** 服务器上 Codex 最初需要人工输入提示词；CLI 版本、审批参数、tmux/cron 环境、代理、认证环境和目标周次不一致时，可能出现连接重试、`CODEX_EXEC_FAILED`、输出周次错误或重复评价。

**调研方法。** 使用只读 preflight、CLI `--help`、显式版本检查、无真实输入的虚构临时目录和 `--dry-run` 分离诊断；不把真实 token、周报或服务器配置放入提示词。把“模型不可用”“输出被阻塞”“输出周次不匹配”“采集失败”分类为安全错误码。

**主要工作。**

- `run_codex_evaluation.py` 负责上一业务周采集、输入指纹、状态锁、临时工作区、隔离 Codex 子进程和原子替换正式报告。
- Harness 在报告覆盖范围、章节、目标周次、敏感字段、状态和大小上做机械校验；不通过校验时保留上一份有效报告。
- 自动评价允许显式私有 HTTPS 代理和 provider/key 配置，但不会直接加载完整 Codex 用户配置或把通用密钥传给模型。
- 增加 `--week-label YYYY-Www`，允许安全地重跑历史 W28；cron 不传参数时仍自动评价上一业务周。
- 同一次 Codex 调用生成正式 `manager_report.md` 和受限的 `employee_feedback.json`，周一通知不再重新解析 Markdown。

**效果。** 评价从“人工启动的长命令”变为可预检、可 dry-run、可重试但不盲目覆盖的自动工作流。Cron 可以在终端关闭后运行，失败状态能定位到具体安全错误码；输入无变化时跳过模型调用，减少 token 消耗。

### 阶段七：前端功能切片与管理交互（2026-07-14 至 07-16）

**问题。** `App.vue` 过大，提交概览和 AI 评价页面信息层级不清；未交卡片、红黑榜、折叠区、负责人统计和 Excel 导出缺少足够可见性；直接把全员周报原文塞入页面还可能造成 `payload too large` 和越权风险。

**主要工作。**

- 抽取认证/API 请求边界后，继续按功能拆分提交概览、AI 评价和授权原文查看；保持现有登录、钉钉 OAuth、角色菜单和克制的登录页视觉。
- 顶部统计统一为应交、已交、未交、负责人；未交卡可进入完整未交列表；周次可展示对应日期；评价规则在页面附近提供说明。
- AI 评价保留正式五维数据，Web 主视图按产品约定展示四维；“本周重点”位于红榜之前，红榜在黑榜之前；员工评价和负责人履职支持收起预览。
- Excel 导出使用标准 OOXML；鼠标悬停时提示浏览器兼容性；单元格按字符串写入，排除 userid、路径、token 和原始周报正文。
- AI 评价中的姓名只基于当前周授权提交状态构建索引，点击后按需读取单人原文；后端服务层再次按 userid/report 过滤，并限制单人预览大小。
- 将姓名提示限制在视口内，避免屏幕最左侧被裁切；移动端保持可用并尊重减少动态效果偏好。

**效果。** AI 页面信息层级更清楚，前端不会批量预取全员正文；原文查看和 Excel 下载都复用后端授权结果，保持其他页面行为不变。

### 阶段八：自动提醒与个人反馈（2026-07-14 至 07-16）

#### 周日 18:00 未交提醒

系统提供可显式启用的周日提醒：只检查当前业务周截至当前时间的提交快照，给缺交人员逐人发送私人钉钉工作通知，同时给管理员发送不含人员明细的执行结果。它是提醒，不是最终逾期结论；状态和不确定发送结果用于阻止自动重复发送。

#### 周一 12:00 个人评价通知

最新功能只面向上一完整业务周的已提交人员：一次评价同时生成私有反馈清单，Harness 要求其稳定 `userid` 集合与已提交集合完全一致；Spring 周一 12:00 按 userid 逐人发送，先肯定证据充分的优点，再突出需要改进的地方，末尾使用服务器私有配置的 HR 联系方式。未提交、免交、评价未就绪、身份不一致或发送状态不确定时，整个批次失败关闭并通知管理员。

两个自动通知默认关闭，所有自动任务固定使用 `Asia/Shanghai`，默认测试不访问真实钉钉。

## 5. 调研与开发方法

### 5.1 先读仓库契约，再看实现

每个风险型任务先阅读根 `AGENTS.md`、产品、架构、质量、安全、Harness 和最近的子目录指南。产品不变量优先于聊天中的临时描述；实现若与契约冲突，先记录证据再做最小修改。

### 5.2 先诊断，再做最小改动

调研通常采用以下顺序：

1. 查看 Git 状态和最近提交，识别他人改动和当前版本。
2. 阅读目标服务、Controller、配置、脚本和前端入口，画出数据边界。
3. 使用虚构数据和临时目录复现确定性逻辑，避免访问真实钉钉、MySQL、`output/`、`logs/` 或 `.env`。
4. 把发现的问题转化为单元测试、HTTP 测试、Harness 检查或受保护哈希。
5. 以最小的可测试重构修复根因，并同步产品/架构/质量/安全文档。
6. 运行定向测试和统一验证，再检查 diff、隐私路径和 Git 状态。

### 5.3 证据分层

- **确定性规则：** Python 和 Spring 单元测试，使用虚构部门、姓名、userid、报告和附件元数据。
- **安全边界：** Spring Security HTTP 测试、服务层过滤测试、错误脱敏测试。
- **外部适配器：** 注入式 HTTP mock，验证请求参数、协议、大小和失败降级，不访问真实钉钉。
- **前端行为：** Vitest/Vue Test Utils mock fetch，验证认证、折叠、导出、姓名交互和错误提示。
- **部署证据：** Compose 配置检查、生产变量校验、服务器侧 preflight/dry-run；真实联通由部署者在私有环境单独确认。

## 6. 技术栈与职责边界

| 层次 | 技术 | 主要职责 | 明确不负责的内容 |
| --- | --- | --- | --- |
| 外部数据 | 钉钉企业应用、通讯录/周报/Drive API、OAuth、工作通知 | 提供授权数据和通知通道 | 不把 token、AppSecret 或原始响应写入仓库/普通日志 |

| 自动评价编排 | Python runtime、状态文件、输入指纹、Cron/CLI | 目标周采集、隔离模型调用、校验和原子替换 | 不在失败或周次错误时覆盖有效报告 |

| 运行与交付 | Docker Compose、Ubuntu、Maven Wrapper、GitHub Actions、Codeup/GitHub | 可重复部署、统一验证和版本协作 | 不将服务器 `.env`、`output/`、`logs/` 纳入版本库 |

### 6.1 当前核心数据流

```text
通讯录 + 周报 API
  -> dingtalk_common / submission_roster
  -> output/<week>/raw、exports、analysis
  -> leader_compliance + 受控附件路径
  -> Codex Skill / Harness
  -> summary/manager_report.md + automation/employee_feedback.json
  -> Spring 服务层按账号范围过滤
  -> Vue 页面、单人原文抽屉、XLSX 下载
```

### 6.2 权限边界

后端服务层决定可见数据；前端只负责展示。`REPORT_ALL` 或显式 `ALL` 才能访问完整周报；`ADMIN` 默认只拥有账号管理和任务相关能力，周报读取仍需范围；普通范围使用 `DEPT:`、`USER:`、`USERID:` 或精确兼容文本。列表、详情、Markdown、CSV、XLSX 和单人原文使用同一服务层过滤。

## 7. Harness 与 Skill 如何配合

### Skill：规定“如何评价”

`weekly-report-assistant` 固化周报分析维度、模板合规与管理评价的分离、负责人额外职责、证据不足时的措辞和正式输出格式。它只接收经过授权和脱敏的输入，不负责账号、权限、采集或通知接收人解析。

### Harness：规定“什么结果才能落盘和发送”

Harness 负责输入指纹、目标周次、临时工作区、子进程隔离、模型输出 Schema、人员覆盖、敏感字段、大小和状态校验。只有通过机械验证的结果才会原子替换正式报告或进入周一通知清单。

### 两者的闭环

```text
确定性采集
  -> Harness 构造最小授权输入
  -> Skill/Codex 生成自然语言评价
  -> Harness 校验周次、章节、覆盖、隐私和大小
  -> 正式报告/私有反馈
  -> Spring 权限过滤或逐人通知
```

这一区分解决了两个常见问题：模型不能替代业务规则和数据权限；程序也不需要用脆弱的字符串规则代替评价判断。

## 9. 关键工程决策

| 决策 | 原因 | 代价/限制 |
| --- | --- | --- |
| 稳定 `userid` 优先，姓名只做精确兼容 | 降低重名和补交错配 | 旧数据若缺 userid 需要待确认 |
| 周四截止归属而非读取正文自述周次 | 周末写作、周中补交，规则可确定 | 需要在跨周边界持续重跑采集 |
| 服务层过滤而非前端隐藏 | 直接 API 调用也必须安全 | 每种输出都要有服务层回归测试 |
| `ADMIN` 与完整报告读取分离 | 账号管理与员工敏感内容职责分离 | 历史固定全权限引导逻辑仍需单独审查 |
| 生产默认拒绝开发凭据 | 避免部署后静默暴露 | 本地必须显式开启开发模式 |
| 原始数据文件化、报告契约化 | 便于 Skill 和 Web 复用，迁移成本低 | 生命周期和审计仍依赖文件目录治理 |
| Skill 负责语义，Harness 负责机械验收 | 兼顾模型灵活性和可控性 | 模型服务失败仍需人工关注状态 |
| 单人原文按需加载 | 降低越权和 payload 风险 | 无法一次性预览全员原文 |
| 通知默认关闭、失败关闭、不确定不重发 | 防止误发和重复通知 | 首次启用需要私有配置和人工联通验证 |

## 10. 当前遗留风险和后续路线

以下内容是已知风险或技术债，不应在介绍项目时写成“已完成”：

1. 历史固定全权限账号引导逻辑仍需独立的产品和安全决策；本阶段没有擅自改变其角色、密码哈希或钉钉绑定流程。
2. 原始周报和负责人附件仍以受控文件保存，需要继续完善保留周期、删除策略和审计记录。
3. 负责人附件下载依赖钉钉 Drive 权限、文件所有者信息和可解析格式，生产权限只能在私有环境验证。
4. Codex 自动评价依赖 CLI 登录、代理、模型额度和服务器网络；preflight 能发现一部分问题，但不能消除外部服务故障。
5. 通知的“远程已接受后进程崩溃”无法在没有外部幂等键的情况下实现严格 exactly-once；当前策略是不确定时阻止自动重发。
6. 前端主包仍较大，`styles.css` 和部分页面仍可继续按功能切片；后续可用路由懒加载和分包降低首屏成本。
7. 周日提醒尚未处理节假日跳过，多实例部署也需要分布式锁或外部任务协调。
8. 未建立真实钉钉/生产环境的自动化端到端测试；生产联通必须由管理员在私有配置和隔离窗口中手工确认。

## 12. 总结

本项目的核心演进不是单纯增加页面或脚本，而是把一个依赖人工经验的周报流程逐层变成可验证的工程系统：

- 用 Python 固化采集、周次、匹配和统计事实；
- 用 Skill 固化评价方法和管理视角；
- 用 Harness 约束模型输出、隐私和自动化失败路径；
- 用 Spring 服务层执行身份、范围和下载权限；
- 用 Vue 提供克制、可解释、按需加载的管理体验；
- 用任务记录、测试、质量预算和安全边界把聊天中的决策沉淀为仓库证据。

因此，项目当前已经从“能拉取并生成周报”发展为“有产品契约、分层架构、回归测试、Harness、自动评价和部署边界的周报工程系统”。后续工作的重点应是控制技术债和提升生产可观测性，而不是再次进行无边界的大规模重写。
